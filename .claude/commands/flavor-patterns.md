## 1. Env class pattern

### Rule: what goes where

| Type | Example | How to store |
|---|---|---|
| Non-secret config | Base URL, app name, feature flags | Hardcoded in Dart `Env` class — safe to commit |
| Secret | Sentry DSN, payment key, analytics ID | `String.fromEnvironment` — injected via `--dart-define` at build time |

**`lib/core/env/env.dart`** — abstract base, no concrete values here:

```dart
/// Abstract environment configuration.
/// Each flavor provides its own concrete implementation.
///
/// Non-secret values are hardcoded per flavor.
/// Secret values use String.fromEnvironment() — passed via --dart-define
/// from CI/CD or from local scripts/run_xxx.sh reading a local .env file.
abstract class Env {
  const Env();

  // ── Non-secret — hardcoded per flavor ───────────────────────────────────
  String get appName;
  String get baseUrl;
  String get flavor;
  bool get isProduction;

  // ── Secrets — injected via --dart-define ─────────────────────────────────
  // Read from compile-time environment. Empty string if not provided.
  // Add/remove secret getters here based on project needs.
  String get sentryDsn => const String.fromEnvironment('SENTRY_DSN');
  String get analyticsId => const String.fromEnvironment('ANALYTICS_ID');

  // Helper: true if a secret was actually provided at build time
  bool get hasSentryDsn => sentryDsn.isNotEmpty;
}
```

**`lib/core/env/env_dev.dart`** — non-secrets hardcoded, secrets via dart-define:

```dart
import 'package:your_app/core/env/env.dart';

class EnvDev extends Env {
  const EnvDev();

  // Non-secret — commit freely
  @override
  String get appName => 'MyApp Dev';

  @override
  String get baseUrl => 'https://dev-api.example.com';

  @override
  String get flavor => 'dev';

  @override
  bool get isProduction => false;

  // Secrets are inherited from Env base class via String.fromEnvironment.
  // No override needed here — they are resolved at compile time from
  // --dart-define=SENTRY_DSN=xxx passed by the run script or CI/CD.
}
```

**`lib/core/env/env_staging.dart`**: same pattern, staging values.

**`lib/core/env/env_production.dart`**: same pattern, `isProduction => true`.

## 2. main_xxx.dart pattern

**`lib/main_dev.dart`**:

```dart
import 'package:flutter/material.dart';
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:your_app/app.dart';
import 'package:your_app/core/env/env_dev.dart';
import 'package:your_app/core/env/env_provider.dart';

void main() {
  runApp(
    ProviderScope(
      overrides: [
        envProvider.overrideWithValue(const EnvDev()),
      ],
      child: const App(),
    ),
  );
}
```

**`lib/main_staging.dart`**: same, `EnvStaging`.
**`lib/main_production.dart`**: same, `EnvProduction`.

**`lib/core/env/env_provider.dart`** — provider that features use:

```dart
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:your_app/core/env/env.dart';

/// Overridden in each main_xxx.dart with the correct Env implementation.
/// Never read before ProviderScope sets the override.
final envProvider = Provider<Env>(
  (ref) => throw UnimplementedError('envProvider must be overridden in main'),
);
```

Usage in any provider/datasource:

```dart
// In DioClient provider
@riverpod
Dio dioClient(Ref ref) {
  final env = ref.read(envProvider);
  final dio = Dio(BaseOptions(baseUrl: env.baseUrl));
  // ...
  return dio;
}
```

## 3. Android build.gradle

Add inside `android { ... }` in `android/app/build.gradle`:

```groovy
flavorDimensions "environment"

productFlavors {
    dev {
        dimension "environment"
        applicationIdSuffix ".dev"
        versionNameSuffix "-dev"
        resValue "string", "app_name", "MyApp Dev"
    }
    staging {
        dimension "environment"
        applicationIdSuffix ".staging"
        versionNameSuffix "-staging"
        resValue "string", "app_name", "MyApp Staging"
    }
    production {
        dimension "environment"
        // No suffix for production
        resValue "string", "app_name", "MyApp"
    }
}
```

Also update `buildTypes` to use `signingConfigs` per flavor if needed:

```groovy
buildTypes {
    release {
        // signingConfig signingConfigs.release
        minifyEnabled true
        proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'), 'proguard-rules.pro'
    }
    debug {
        // Default debug signing
    }
}
```

## 4. Android strings.xml per flavor

Each flavor needs `android/app/src/{flavor}/res/values/strings.xml`
**only if** overriding string resources per flavor. The `app_name` is usually
handled via `resValue` in `build.gradle` above so this file may be empty.

```xml
<?xml version="1.0" encoding="utf-8"?>
<resources>
    <!-- Flavor-specific strings go here if needed -->
    <!-- app_name is set via resValue in build.gradle -->
</resources>
```

**Google Services placeholder** `android/app/src/{flavor}/google-services.json`:

```json
{
  "_comment": "TODO: Replace with your actual google-services.json from Firebase Console",
  "_flavor": "dev",
  "project_info": {
    "project_id": "your-project-dev"
  }
}
```

## 5. iOS xcconfig

**`ios/Flutter/dev.xcconfig`**:

```
#include "Generated.xcconfig"
FLUTTER_TARGET=lib/main_dev.dart
BUNDLE_ID_SUFFIX=.dev
APP_NAME=MyApp Dev
API_BASE_URL=https://dev-api.example.com
```

**`ios/Flutter/staging.xcconfig`**:

```
#include "Generated.xcconfig"
FLUTTER_TARGET=lib/main_staging.dart
BUNDLE_ID_SUFFIX=.staging
APP_NAME=MyApp Staging
API_BASE_URL=https://staging-api.example.com
```

**`ios/Flutter/production.xcconfig`**:

```
#include "Generated.xcconfig"
FLUTTER_TARGET=lib/main_production.dart
BUNDLE_ID_SUFFIX=
APP_NAME=MyApp
API_BASE_URL=https://api.example.com
```

> **Note:** These xcconfig values are only for reference in Xcode schemes.
> The Dart `Env` class is the single source of truth at runtime.
> iOS `Info.plist` can read `$(APP_NAME)` from xcconfig if configured in Xcode.

## 6. Run scripts

Scripts load a local `.env` file (if present) and forward each key as `--dart-define`.
The `.env` file is **never committed** — only used for local development convenience.

**`.env.example`** — commit this as documentation of available keys:

```bash
# Copy to .env and fill in your local values.
# .env is gitignored — never commit real secrets.
SENTRY_DSN=
ANALYTICS_ID=
PAYMENT_KEY=
```

**`.env`** — local only, gitignored, filled in by each developer:

```bash
SENTRY_DSN=https://xxx@sentry.io/123
ANALYTICS_ID=G-XXXXXXXX
PAYMENT_KEY=sk_test_xxx
```

**`scripts/_load_env.sh`** — shared helper, sourced by each run script:

```bash
#!/bin/bash
# Loads .env file and converts each line to --dart-define=KEY=VALUE args.
# Usage: source scripts/_load_env.sh
#        then use $DART_DEFINES in flutter run/build commands.

DART_DEFINES=""

if [ -f ".env" ]; then
  while IFS='=' read -r key value; do
    # Skip comments and empty lines
    [[ "$key" =~ ^#.*$ || -z "$key" ]] && continue
    # Trim whitespace
    key=$(echo "$key" | xargs)
    value=$(echo "$value" | xargs)
    [ -n "$key" ] && DART_DEFINES="$DART_DEFINES --dart-define=$key=$value"
  done < .env
  echo "✓ Loaded $(grep -c '^[^#]' .env) keys from .env"
else
  echo "⚠ No .env file found — secrets will be empty strings"
fi
```

**`scripts/run_dev.sh`**:

```bash
#!/bin/bash
set -e
source "$(dirname "$0")/_load_env.sh"

echo "▶ Running DEV flavor..."
flutter run \
  --flavor dev \
  -t lib/main_dev.dart \
  $DART_DEFINES \
  "$@"
```

**`scripts/run_staging.sh`**:

```bash
#!/bin/bash
set -e
source "$(dirname "$0")/_load_env.sh"

echo "▶ Running STAGING flavor..."
flutter run \
  --flavor staging \
  -t lib/main_staging.dart \
  $DART_DEFINES \
  "$@"
```

**`scripts/run_production.sh`**:

```bash
#!/bin/bash
set -e
source "$(dirname "$0")/_load_env.sh"

echo "▶ Running PRODUCTION flavor..."
flutter run \
  --flavor production \
  -t lib/main_production.dart \
  $DART_DEFINES \
  "$@"
```

Make executable after creating:
```bash
chmod +x scripts/_load_env.sh \
         scripts/run_dev.sh \
         scripts/run_staging.sh \
         scripts/run_production.sh
```

**CI/CD** (GitHub Actions example) — no `.env` file, uses repo secrets directly:

```yaml
- name: Build production APK
  run: |
    flutter build apk \
      --flavor production \
      -t lib/main_production.dart \
      --dart-define=SENTRY_DSN=${{ secrets.SENTRY_DSN }} \
      --dart-define=ANALYTICS_ID=${{ secrets.ANALYTICS_ID }}
```

## 7. VS Code launch.json

VS Code không đọc `.env` file tự động — cần khai báo secrets trực tiếp trong `toolArgs`.
Dùng `envFile` để VS Code load `.env` và truyền vào `--dart-define`:

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Dev",
      "request": "launch",
      "type": "dart",
      "program": "lib/main_dev.dart",
      "args": ["--flavor", "dev"],
      "toolArgs": [
        "--dart-define=SENTRY_DSN=${env:SENTRY_DSN}",
        "--dart-define=ANALYTICS_ID=${env:ANALYTICS_ID}"
      ]
    },
    {
      "name": "Staging",
      "request": "launch",
      "type": "dart",
      "program": "lib/main_staging.dart",
      "args": ["--flavor", "staging"],
      "toolArgs": [
        "--dart-define=SENTRY_DSN=${env:SENTRY_DSN}",
        "--dart-define=ANALYTICS_ID=${env:ANALYTICS_ID}"
      ]
    },
    {
      "name": "Production",
      "request": "launch",
      "type": "dart",
      "program": "lib/main_production.dart",
      "args": ["--flavor", "production"],
      "toolArgs": [
        "--dart-define=SENTRY_DSN=${env:SENTRY_DSN}",
        "--dart-define=ANALYTICS_ID=${env:ANALYTICS_ID}"
      ]
    }
  ]
}
```

> `${env:KEY}` đọc từ shell environment. Để VS Code tự load `.env` khi launch,
> cài extension **DotENV** và thêm vào `.vscode/settings.json`:
> ```json
> { "dart.env": { "SENTRY_DSN": "" } }
> ```
> Hoặc đơn giản hơn: dùng `scripts/run_xxx.sh` từ terminal thay vì VS Code debugger
> khi cần secrets — script đã tự load `.env`.
```

## 8. flutter_launcher_icons config

Add to `pubspec.yaml` (requires `flutter_launcher_icons: ^0.14.0` in `dev_dependencies`):

```yaml
flutter_launcher_icons:
  image_path: "assets/icon/icon.png"
  android: true
  ios: true

flutter_launcher_icons_dev:
  image_path: "assets/icon/icon_dev.png"
  android: "ic_launcher"
  ios: true
  flavor: dev

flutter_launcher_icons_staging:
  image_path: "assets/icon/icon_staging.png"
  android: "ic_launcher"
  ios: true
  flavor: staging

flutter_launcher_icons_production:
  image_path: "assets/icon/icon.png"
  android: "ic_launcher"
  ios: true
  flavor: production
```

Run after configuring:
```bash
dart run flutter_launcher_icons -f pubspec.yaml
```

> The simplest approach for dev/staging badges: use a different icon image with a
> colored banner overlay (DEV / STG) baked into `icon_dev.png` / `icon_staging.png`.
> Tools like [icon_kitchen](https://icon.kitchen) can generate these quickly.

## 9. Riverpod DI — how Env is injected

The `envProvider` is overridden at the root `ProviderScope` in each `main_xxx.dart`.
All other providers that need env config read it via `ref.read(envProvider)`.

```
main_dev.dart
  └── ProviderScope(overrides: [envProvider ← EnvDev()])
        └── dioClientProvider
              └── ref.read(envProvider).baseUrl  ← "https://dev-api.example.com"

main_production.dart
  └── ProviderScope(overrides: [envProvider ← EnvProduction()])
        └── dioClientProvider
              └── ref.read(envProvider).baseUrl  ← "https://api.example.com"
```

Never hardcode `EnvDev()` or `EnvProduction()` inside providers or datasources —
always go through `ref.read(envProvider)`.
