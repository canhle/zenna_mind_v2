## CRITICAL RULE

The plan file `docs/design/$ARGUMENTS/ui_plan.md` is the **SINGLE SOURCE OF TRUTH**.

- Every class name, field type, file path, component structure, l10n key, and spacing value is defined in the plan.
- This command file only defines the **execution procedure** (what order to do things, what commands to run).
- When in doubt, re-read the plan — do NOT invent details or fall back to generic patterns.
- Do NOT hardcode any UI text, spacing values, or style choices — always look them up in the plan.

---

Read these files in order:

1. `docs/design/$ARGUMENTS/ui_plan.md` — **read fully, do not skim**
2. `docs/coding-conventions.md`
3. `docs/coding-checklist.md`

If the plan file does not exist, STOP and tell the user to run `/feature-ui-plan $ARGUMENTS` first.

### Implementation rules (apply to every file):

- **Use exact names from the plan** — class names, file names, field names, l10n keys
- **Use exact values from the plan** — spacing (`DsSpacing.*`), radius (`DsRadius.*`), colors (`DsColors.*`), text styles (`Theme.of(context).textTheme.*`)
- **Use exact component structure from the plan** — which widget watches which field via `.select()`
- **Use design system tokens** — never hardcode hex colors, pixel spacing, or font styles
- Read `lib/design_system/components/` before creating new widgets — reuse existing DS components per plan §4.3
- Read `lib/core/widgets/` for `LoadingView` and `ErrorView`
- **ViewModel mixin:** use `with ListenWithAutoClose` (from `core/mixins/listen_with_auto_close.dart`) when using `listenWithAutoClose`
- **L10n access:** `S.of(context)!.keyName` in widgets
