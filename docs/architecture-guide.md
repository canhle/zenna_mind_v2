# Architecture Guide

Flutter Clean Architecture template using Riverpod, Freezed, and Cloud Firestore as the primary data backend.

> **Data backend**: This project reads/writes data directly from Cloud Firestore. The schema is defined in `docs/db/zenna_mind_database_design.pdf` — that document is the authoritative source for all collection paths and document shapes. `dio` is kept in the codebase for potential future REST endpoints, but no current feature uses it.

## Overview

This template follows **Clean Architecture** with a hybrid folder organization:

- `domain/` and `data/` are organized **layer-first** — shared entities and repositories live in one place, accessible across all features.
- `features/` (presentation) is organized **feature-first** — each screen owns its ViewModel, UiState, and components.

**Core stack:**

| Concern | Library |
|---|---|
| State management | `riverpod` + `riverpod_annotation` |
| Immutable state | `freezed` |
| Data backend | `cloud_firestore` (primary) |
| Auth | `firebase_auth` |
| HTTP (reserved) | `dio` — kept for future REST use, not wired to any feature today |
| Navigation | `go_router` |
| Storage | `shared_preferences` + `flutter_secure_storage` |
| Localization | Flutter `gen-l10n` (ARB) |
| Code generation | `build_runner`, `freezed_generator`, `riverpod_generator` |

## Layer Responsibilities

### `design_system/`

The app's UI kit — maps 1:1 to the Figma component library. No business logic, no app context. Only knows about design tokens.

- **tokens/** — design primitives from Figma: `DsColors`, `DsTypography`, `DsSpacing`, `DsRadius`, `DsShadows`, `DsDurations`.
- **components/** — reusable UI building blocks: `DsButton`, `DsTextField`, `DsBadge`, `DsCard`, `DsBottomSheet`, `DsDialog`, `DsAppBar`, `DsToast`. These accept only primitive inputs (`String`, `VoidCallback`) — never entities or UiState.
- **icons/** — custom icon set exported from Figma (`DsIcons`).
- **theme/** — `DsTheme` maps tokens to `ThemeData` (light/dark).
- **design_system.dart** — barrel export for single import.

### `core/`

Shared app infrastructure used by all layers. No business logic. Imports nothing from domain/data/features. May import from `design_system/`.

- **constants/** — pagination sizes, scroll thresholds.
- **env/** — flavor configuration (dev/staging/production base URLs, app names).
- **error/** — sealed `Failure` hierarchy used as the error language across all layers.
- **extensions/** — convenience extensions for **SDK types only** (`BuildContext`, `String`, `DateTime`). Domain entity extensions belong in `domain/entities/`.
- **l10n/** — ARB translation files and auto-generated `S` class. Keys grouped by feature prefix (`common_`, `error_`, `productList_`, `auth_`).
- **mixins/** — `ListenWithAutoClose` mixin for ViewModel data loading with auto-cancellation.
- **firebase/** — `FirebaseFirestore` instance provider. Features read/write through this.
- **network/** — Dio client, interceptors, and centralized endpoint URLs. **Reserved for future REST use**; no current feature depends on this.
- **router/** — GoRouter configuration with `AppRoutes` constants.
- **services/** — persistent storage (SharedPreferences for prefs, SecureStorage for tokens).
- **utils/** — `Debouncer`, `AppLogger`, `Validators`.
- **widgets/** — app-specific widgets (`LoadingView`, `ErrorView`) that **use design_system components** internally but know about app context (localization, error types).

### `domain/`

The heart of the app. No Flutter imports, no `cloud_firestore` imports, no Dio imports — pure Dart only.

- **Entities** are the shared business objects. Because entities are used across multiple features (a `Product` appears in product list, cart, and checkout), they live here rather than inside any single feature folder. Entity extensions (e.g. `isOutOfStock`, `formattedPrice`) live next to the entity file.
- **Repository interfaces** define what data operations are available without specifying how they are performed.
- **UseCases** are always created for every feature to ensure consistency. They contain business rules: input validation, filtering, data transformation, coordination across repositories. Even when a UseCase only forwards to a repository, it provides a ready place to add logic later.
- **Domain providers** wire UseCases into Riverpod and expose functional providers (`productsProvider`) that ViewModels listen to.

### `data/`

Implementation of the domain contracts. Knows about Firestore documents, JSON, and caching. Does not know about UI.

- **Models** mirror Firestore document shapes. They provide `fromFirestore(DocumentSnapshot)` and `toFirestore()` helpers (or `@JsonSerializable` + manual map conversion). A `toEntity()` method converts them to domain entities, applying any field renaming, type parsing, and filtering.
- **DataSources** query Cloud Firestore via `FirebaseFirestore.instance`. They are stateless — no caching, no retry, no business logic. Each DataSource contains a single `try/catch` boundary that maps `FirebaseException` → `Failure` (Firestore has no interceptor mechanism, so error translation lives at the call site, wrapped in a shared helper).
- **RepositoryImpl** implements the domain interface. Its job is to call the DataSource and map Models to Entities.
- **Data providers** wire DataSources and RepositoryImpl into Riverpod.

> Note: Dio-based REST DataSources are still legal and may be added later. They would use `ErrorInterceptor` for `DioException` → `Failure` translation. Both error boundaries map to the same `Failure` hierarchy.

### `features/`

Everything the user sees and interacts with.

- **Screen** is a `ConsumerStatefulWidget` that reads UiState and dispatches events to the ViewModel.
- **ViewModel** is a `Notifier<UiState>`. It listens to domain providers via `listenWithAutoClose`, handles errors, and emits Events.
- **UiState** is a `@freezed` class that captures the complete snapshot of a screen at any moment — async data fields, UI flags, and a nullable Event.
- **Components** are sub-widgets extracted from the screen for rebuild isolation and reuse.

## Data Flow

### Happy path (fetch products)

```
ProductListScreen
  └── ref.watch(productListViewModelProvider(args))
        └── ProductListUiState { products: AsyncData(ProductsData(...)) }

ProductListViewModel.fetchProducts()
  └── listenWithAutoClose(productsProvider(...))
        └── productsProvider (domain)
              └── GetProductsUseCase.call(...)
                    └── ProductRepository.getProducts(...)  [abstract]
                          └── ProductRepositoryImpl.getProducts(...)
                                └── ProductFirestoreDataSource.getProducts(...)
                                      └── FirebaseFirestore.instance
                                            .collection('products').get()
```

### Error path

```
FirebaseFirestore.collection('products').get() → FirebaseException
  └── ProductFirestoreDataSource catch block (single try/catch boundary)
        ├── code 'unavailable' / network issue → NetworkFailure
        ├── code 'deadline-exceeded' → TimeoutFailure
        ├── code 'permission-denied' → ForbiddenFailure
        ├── code 'unauthenticated' → UnauthorizedFailure
        ├── code 'not-found' → NotFoundFailure
        └── other → ServerFailure(errorCode: exception.code)
              ↓
        productsProvider → AsyncError(Failure)
              ↓
        ProductListViewModel._handleError()
              ↓
        state.copyWith(event: ShowError(...))
              ↓
        ProductListScreen ref.listen → SnackBar / context.go(...)
```

## Key Design Decisions

### Why `Notifier<UiState>` instead of `AsyncNotifier`?

`AsyncNotifier` forces the entire ViewModel state to be a single `AsyncValue<T>`. This makes it impossible to hold multiple independent async fields, UI flags, or Events alongside the primary data. A synchronous `Notifier<UiState @freezed>` keeps the full state snapshot visible and testable.

```dart
// AsyncNotifier — entire state is AsyncValue<T>
// Cannot add selectedId, isSubmitting, or event without a wrapper hack
class ProductListNotifier extends AsyncNotifier<List<Product>> { }

// Notifier<UiState> — complete, extensible snapshot
@freezed
class ProductListUiState with _$ProductListUiState {
  const factory ProductListUiState({
    @Default(AsyncValue.loading()) AsyncValue<ProductsData> products,
    @Default(false) bool isSubmitting,
    ProductListEvent? event,
  }) = _ProductListUiState;
}
```

### Why no Riverpod Hooks?

`flutter_hooks` and Riverpod hooks (`HookConsumerWidget`) mix two paradigms — class-based OOP and functional composition. For a mixed-level team, this creates unnecessary cognitive overhead. `ConsumerWidget` and `ConsumerStatefulWidget` cover all real-world cases clearly.

### Why hybrid folder organization?

Pure feature-first breaks down when entities are shared across features. A `Product` entity used in product list, cart, and checkout has no natural "home" feature. Placing all entities in `domain/entities/` gives them a single canonical location. Presentation stays feature-first because that is the layer developers navigate most frequently.

### Why only one error layer (`Failure`)?

Maintaining `AppException` (data layer) and `Failure` (domain layer) as two separate hierarchies only adds a translation step with no real benefit — it is just a rename. Instead, each DataSource translates its infrastructure exception directly to `Failure` at its own boundary.

- **FirestoreDataSource** wraps its queries in a `try/catch` and maps `FirebaseException.code` → `Failure`.
- **(Future) Dio-based DataSource** would let `ErrorInterceptor` translate `DioException` → `Failure` automatically.

Both paths converge on the same `Failure` hierarchy, so UseCases, Repositories, and ViewModels never need to know which transport produced the error.

### Why pass only IDs between screens?

Passing a full entity from a list screen to a detail screen means the detail screen shows potentially stale data. The detail screen should always fetch its own fresh copy using the ID.

### Why `core/extensions/` only for SDK types?

Extensions for domain entities (e.g. `Product.isOutOfStock`) are business logic — they belong in `domain/entities/` next to the entity. Placing them in `core/extensions/` would make `core/` depend on `domain/`, violating the dependency rule. Feature-specific extensions stay in the feature folder and get promoted when shared.

### Why ARB over JSON for localization?

ARB is Flutter's official format with built-in `gen-l10n` support, type-safe generated code, and native plural/gender/select support. No third-party package needed. Keys are grouped by feature prefix (`productList_title`, `auth_login`) to keep the flat namespace organized.

### Why `design_system/` separate from `core/widgets/`?

They serve different purposes:

- **`design_system/`** — the app's UI kit (maps to Figma). Components only accept primitive inputs (`String label`, `VoidCallback onPressed`). They know nothing about business logic. Could theoretically be reused in another app with the same design.
- **`core/widgets/`** — app-specific composite widgets that **use** design_system components internally but know about app context (localization, error types, retry patterns).

Dependency flow: `features/ → core/widgets/ → design_system/`

## Creating a New Feature — Step by Step

Use the `product` feature as a reference. Replace `product` / `Product` with your feature name.

### 1. Domain — Entity

```dart
// domain/entities/order.dart
@freezed
class Order with _$Order {
  const factory Order({
    required String id,
    required List<OrderItem> items,
    required OrderStatus status,
    required DateTime createdAt,
  }) = _Order;
}
```

### 2. Domain — Repository interface

```dart
// domain/repositories/order_repository.dart
abstract interface class OrderRepository {
  Future<List<Order>> getOrders({required int page});
  Future<Order> getOrderDetail({required String orderId});
}
```

### 3. Domain — UseCase (only if business logic exists)

```dart
// domain/usecases/order/get_orders_usecase.dart
class GetOrdersUseCase {
  const GetOrdersUseCase(this._repository);
  final OrderRepository _repository;

  Future<List<Order>> call({required int page}) async {
    if (page < 0) throw ArgumentError('Page must be non-negative');
    final orders = await _repository.getOrders(page: page);
    return orders.where((o) => o.status != OrderStatus.cancelled).toList();
  }
}
```

### 4. Data — Model

```dart
// data/models/order_model.dart
@JsonSerializable()
class OrderModel {
  final String orderId;
  final String statusCode;
  final String createdAt;

  factory OrderModel.fromJson(Map<String, dynamic> json) =>
      _$OrderModelFromJson(json);

  Order toEntity() => Order(
    id: orderId,
    items: [],
    status: OrderStatus.fromString(statusCode),
    createdAt: DateTime.parse(createdAt),
  );
}
```

### 5. Data — DataSource (Firestore)

```dart
// data/datasources/order_firestore_datasource.dart
abstract interface class OrderFirestoreDataSource {
  Future<List<OrderModel>> getOrders({required String userId, required int limit});
}

class OrderFirestoreDataSourceImpl implements OrderFirestoreDataSource {
  const OrderFirestoreDataSourceImpl(this._firestore);
  final FirebaseFirestore _firestore;

  @override
  Future<List<OrderModel>> getOrders({
    required String userId,
    required int limit,
  }) async {
    try {
      // Firestore path per docs/db/zenna_mind_database_design.pdf
      final snapshot = await _firestore
          .collection('users')
          .doc(userId)
          .collection('orders')
          .orderBy('createdAt', descending: true)
          .limit(limit)
          .get();

      return snapshot.docs.map(OrderModel.fromFirestore).toList();
    } on FirebaseException catch (e) {
      throw _mapFirestoreError(e);
    }
  }
}

// Shared helper used by all Firestore DataSources
Failure _mapFirestoreError(FirebaseException e) => switch (e.code) {
      'unavailable' || 'cancelled' => const NetworkFailure(),
      'deadline-exceeded' => const TimeoutFailure(),
      'permission-denied' => const ForbiddenFailure(),
      'unauthenticated' => const UnauthorizedFailure(),
      'not-found' => const NotFoundFailure(),
      _ => ServerFailure(errorCode: e.code),
    };
```

### 6. Data — RepositoryImpl + providers

```dart
// data/repositories/order_repository_impl.dart
class OrderRepositoryImpl implements OrderRepository {
  const OrderRepositoryImpl(this._dataSource);
  final OrderFirestoreDataSource _dataSource;

  @override
  Future<List<Order>> getOrders({required String userId, required int limit}) async {
    final models = await _dataSource.getOrders(userId: userId, limit: limit);
    return models.map((m) => m.toEntity()).toList();
  }
}

// data/providers/order_data_providers.dart
@riverpod
OrderFirestoreDataSource orderFirestoreDataSource(Ref ref) =>
    OrderFirestoreDataSourceImpl(ref.read(firestoreProvider));

@riverpod
OrderRepository orderRepository(Ref ref) =>
    OrderRepositoryImpl(ref.read(orderFirestoreDataSourceProvider));
```

### 9. Presentation — ViewModel

```dart
// features/order_list/order_list_view_model.dart
@riverpod
class OrderListViewModel extends _$OrderListViewModel
    with ListenWithAutoClose {
  @override
  OrderListUiState build(OrderListArguments args) {
    return const OrderListUiState();
  }

  void fetchOrders() {
    state = state.copyWith(orders: const AsyncLoading());

    listenWithAutoClose<AsyncValue<List<Order>>>(
      provider: ordersProvider(page: 0),
      ref: ref,
      onValue: (value) => value.when(
        data: (orders) => state = state.copyWith(
          orders: AsyncData(OrdersData(items: orders)),
        ),
        loading: () {},
        error: (e, _) => _handleError(e),
      ),
    );
  }

  void consumeEvent() => state = state.copyWith(event: null);

  void _handleError(Object error) {
    final message = switch (error) {
      UnauthorizedFailure() => S.current.error_unauthorized,
      NetworkFailure() => S.current.error_network,
      TimeoutFailure() => S.current.error_network,
      ServerFailure() => S.current.error_server,
      _ => S.current.error_unknown,
    };

    if (state.orders.valueOrNull == null) {
      state = state.copyWith(
        orders: AsyncError(error, StackTrace.current),
        event: ShowError(message),
      );
    } else {
      state = state.copyWith(event: ShowError(message));
    }
  }
}
```

### 10. Presentation — Screen

```dart
// features/order_list/order_list_screen.dart
class OrderListScreen extends ConsumerStatefulWidget {
  const OrderListScreen({super.key, required this.args});
  final OrderListArguments args;

  @override
  ConsumerState<OrderListScreen> createState() => _OrderListScreenState();
}

class _OrderListScreenState extends ConsumerState<OrderListScreen> {
  @override
  void initState() {
    super.initState();
    Future.microtask(() =>
      ref.read(orderListViewModelProvider(widget.args).notifier)
          .fetchOrders(),
    );
  }

  @override
  Widget build(BuildContext context) {
    // Handle one-shot events
    ref.listen(orderListViewModelProvider(widget.args), (_, state) {
      final event = state.event;
      if (event == null) return;

      switch (event) {
        case NavigateToDetail(:final orderId):
          context.push('/orders/$orderId');
        case ShowError(:final message):
          ScaffoldMessenger.of(context)
              .showSnackBar(SnackBar(content: Text(message)));
      }

      ref.read(orderListViewModelProvider(widget.args).notifier)
          .consumeEvent();
    });

    final orders = ref.watch(
      orderListViewModelProvider(widget.args).select((s) => s.orders),
    );

    return Scaffold(
      appBar: AppBar(title: const Text('Orders')),
      body: orders.when(
        data: (data) => _OrderListView(orders: data.items, args: widget.args),
        loading: () => const LoadingView(),
        error: (e, _) => ErrorView(
          onRetry: () => ref
              .read(orderListViewModelProvider(widget.args).notifier)
              .fetchOrders(),
        ),
      ),
    );
  }
}
```
